## 1. Current State

### 1.1 Existing Code Analysis

The LTI system is a full-stack recruitment candidate management application:

- **Backend**: Express + TypeScript + Prisma ORM + PostgreSQL
- **Frontend**: React (Create React App) - partially implemented
- **Port**: API on `:3010`, Frontend on `:3000`

**Detected Architecture:**
```
Controller (candidates) → Service (candidateService) → Domain Model (Candidate) → Prisma
```

**Domain Entities:**
- `Candidate` (id, firstName, lastName, email, phone, address)
- `Education` (institution, title, startDate, endDate, candidateId)
- `WorkExperience` (company, position, description, startDate, endDate, candidateId)
- `Resume` (filePath, fileType, uploadDate, candidateId)

### 1.2 Documented Business Rules

## 3. Functional Requirements

### 3.1 Atomic Acceptance Criteria

**Candidate Creation:**

- [ ] POST /candidates with valid data returns 201 and candidate with assigned ID
- [ ] Duplicate email returns 400 with message "The email already exists"
- [ ] Empty firstName returns 400 with "Invalid name"
- [ ] lastName < 2 characters returns 400
- [ ] Invalid email returns 400 "Invalid email"
- [ ] Non-Spanish phone returns 400 "Invalid phone"
- [ ] Education without institution returns 400
- [ ] WorkExperience without company returns 400
- [ ] Invalid date returns 400 "Invalid date"

**CV Upload:**

- [ ] POST /upload with PDF returns 200 with filePath and fileType
- [ ] POST /upload with DOCX returns 200
- [ ] POST /upload with image returns 400 "Invalid file type"
- [ ] POST /upload with file > 10MB returns 500 Multer error
- [ ] File saved to ../uploads/ with timestamp prefix

**Business Rules Validation:**

- [ ] endDate before startDate in Education is REJECTED (or warning)
- [ ] endDate before startDate in WorkExperience is REJECTED (or warning)
- [ ] Edit mode (with ID) allows optional fields
- [ ] Updating existing resume is NOT allowed

### 3.2 Derived Use Cases (for agent)

1. **UC001**: As a recruiter, I want to register a candidate with their basic info, education, and work experience
2. **UC002**: As a recruiter, I want to upload a CV in PDF or DOCX format
3. **UC003**: As a recruiter, I want to see the list of registered candidates
4. **UC004**: As a recruiter, I want to search for a candidate by email
5. **UC005**: As a recruiter, I want to update a candidate's data
6. **UC006**: As a recruiter, I want to delete a candidate

**Detected in current code:**

| Item | State | Action |
|------|-------|--------|
| CORS hardcoded | ⚠️ Risk | Implement configurable whitelist |
| No authentication | ❌ Does not exist | Implement JWT or NextAuth |

| SQL Injection (Prisma protects) | ✅ Secure | N/A |

- [ ] JWT-based authentication with refresh tokens

## 6. Notes for Agent

### 6.1 Conflicts Detected

The following points require clarification before proceeding:

1. **Frontend Implementation**: Current code only has React skeleton (CRA boilerplate). Implement full UI or only refactor existing AddCandidateForm?

2. **Single Resume vs Multiple**: Current logic allows only ONE CV per candidate. Is this a business requirement or a bug to fix?

3. **Authentication**: No auth system exists. Implement JWT or is it for internal-only use without auth?

4. **Delete Strategy**: FK with ON DELETE RESTRICT means candidate with related records cannot be deleted. Cascade delete or soft delete?

5. **Email as Unique**: Only one email per candidate. Should multiple emails be supported?

### 6.2 Assumptions Made

To generate this PRD, the following defaults were assumed (validate with PO):

- System is for internal recruiter use
- Multi-tenant not required
- Only human candidates (not companies)
- Spanish as UI language
- Deployment on-premise or cloud (not specified)
